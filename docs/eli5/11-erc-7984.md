# ERC-7984 — Confidential Tokens (ELI5)

> **TL;DR** — ERC-7984 is a token standard that works like a regular ERC-20, except all balances and transfer amounts are encrypted. You can still send, receive, and approve tokens — you just can't spy on anyone else's balance.

---

## The Sealed Envelope Analogy

Imagine regular ERC-20 tokens as postcards — everyone at the post office can read the message ("Alice has 1000 USDC").

ERC-7984 tokens are **sealed envelopes**. The postcard system still works the same way (same addresses, same delivery routes), but the contents are sealed. Only the sender and recipient can open their own envelopes.

| Feature | ERC-20 (postcard) | ERC-7984 (sealed envelope) |
|---|---|---|
| Balance visibility | Public — anyone can call `balanceOf()` | Encrypted — `confidentialBalanceOf()` returns a handle |
| Transfer amounts | Visible in tx data and events | Encrypted — amounts are handles |
| Approval mechanism | Public `approve(spender, amount)` | Encrypted approval amounts |
| Total supply | Public | Can be public or encrypted (configurable) |
| Wallet compatibility | All wallets | All wallets (no special wallet needed) |

---

## What Makes ERC-7984 Special

### 1. It's an ERC-20 Extension, Not a Replacement

ERC-7984 doesn't throw away ERC-20. It **extends** it. A confidential token contract implements:

- The standard ERC-20 interface (for compatibility with wallets, explorers, DEXes)
- Additional confidential functions for encrypted operations

This means:
- MetaMask can display the token (it sees it as an ERC-20)
- Etherscan can list it (it recognizes the standard events)
- DeFi protocols can integrate it (it speaks the ERC-20 language)

### 2. Balances Are Handles

In a regular ERC-20:
```solidity
mapping(address => uint256) public balanceOf;
// Anyone can read: balanceOf[alice] = 1000
```

In ERC-7984:
```solidity
mapping(address => euint256) private _balances;
// balanceOf[alice] = 0x7f3a...b2c1 (a handle — means nothing without decryption)
```

The handle is a 32-byte pointer to an encrypted value stored off-chain. To see the actual balance, you must be authorized (owner or viewer) and decrypt it via the SDK.

### 3. Native Composability

Because ERC-7984 tokens are valid ERC-20 tokens, they can participate in the DeFi ecosystem:

- **DEXes** can list them
- **Lending protocols** can use them as collateral
- **Yield vaults** can hold them
- **Bridges** can transfer them across chains

The confidential layer is **additive** — it doesn't break existing integrations.

## The Core Functions

### Reading Balances

```solidity
// ERC-20: returns a readable number
function balanceOf(address account) external view returns (uint256);

// ERC-7984: returns an encrypted handle
function confidentialBalanceOf(address account) external view returns (euint256);
```

To decrypt the handle:
```typescript
const handle = await contract.confidentialBalanceOf(userAddress);
const { value } = await handleClient.decrypt(handle);
// value = 1000n
```

### Transfers

```solidity
// ERC-20: amount is visible on-chain
function transfer(address to, uint256 amount);

// ERC-7984: amount is encrypted
function confidentialTransfer(
    address to,
    externalEuint256 encryptedAmount,
    bytes calldata proof
);
```

Under the hood, `confidentialTransfer` uses Nox's atomic `transfer` operation, which updates both the sender's and receiver's balance handles in a single computation.

### Wrapping and Unwrapping

ERC-7984 tokens typically come with a **wrapper** that converts between public and confidential versions:

```
Public USDC ──── wrap() ────> cUSDC (confidential)
                                │
cUSDC (confidential) ── unwrap() ──> Public USDC
```

- **Wrap**: Lock public tokens in the contract, mint equivalent confidential tokens (1:1 ratio)
- **Unwrap**: Burn confidential tokens, release public tokens back to the user

## How a Confidential Transfer Works

Let's trace what happens when Alice sends 100 cUSDC to Bob:

```
1. Alice encrypts "100" via SDK → gets handle_100 + proof

2. Alice calls confidentialTransfer(bobAddress, handle_100, proof)

3. Smart contract:
   a. Verify proof → Nox.fromExternal(handle_100, proof)
   b. Atomic transfer → Nox.transfer(handle_100, aliceBalance, bobBalance)
      This emits an event with 3 input handles and 3 output handles

4. Off-chain Runner (in TEE):
   a. Decrypt all inputs: amount=100, aliceBalance=500, bobBalance=200
   b. Compute: if amount ≤ aliceBalance → alice=400, bob=300
   c. Encrypt results → store behind new handles

5. On-chain result:
   - Alice's balance handle: new_handle_alice (decrypts to 400)
   - Bob's balance handle: new_handle_bob (decrypts to 300)
   - Nobody on-chain knows the amounts changed or what they are
```

## Key Takeaways

1. **ERC-7984 = ERC-20 + encrypted balances.** It extends the standard rather than replacing it, so existing tools and protocols keep working.

2. **Balances are handles**, not numbers. You need permission and the SDK to see actual values.

3. **Transfers are atomic.** The `transfer` operation updates sender and receiver balances in one step, inside the TEE.

4. **Wrap/Unwrap bridges the gap** between public ERC-20 tokens and their confidential ERC-7984 counterparts.

5. **No special wallet required.** MetaMask, Rabby, Coinbase Wallet — they all work because ERC-7984 speaks ERC-20.
